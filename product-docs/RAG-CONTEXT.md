## 5. Generation Pipeline (DEEP — alur end-to-end)

Sumber: `src/app/api/v1/generate/route.ts` (564 baris) + `src/lib/ai/llm-client.ts` (424 baris).

### Step-by-step flow

1. **POST masuk** (`route.ts:53`). `runtime='nodejs'`, `maxDuration=300`, `force-dynamic` (`route.ts:19-21`).
2. **Auth**: `auth()` NextAuth (`route.ts:59`); 401 jika no session (`route.ts:64`).
3. **Body parse**: `req.json()` → `GenerateInputSchema.safeParse` (`route.ts:70-75`). 400/422 jika invalid (`route.ts:80-88`).
4. **Provider resolve**: by `providerId` or active or fallback first (`route.ts:92-99`). 404 jika null (`route.ts:105-108`). Validasi URL (`route.ts:115-120`).
5. **Project resolve**: create new (`createProject`, `route.ts:125-135`) atau verify ownership (`getProjectById`, `route.ts:137-139`).
6. **Orphan refs attach** (V2, `route.ts:148-159`) — non-fatal.
7. **SSE stream start** (`route.ts:163-554`). Headers: `text/event-stream`, `no-cache`, `X-Accel-Buffering: no` (`route.ts:557-563`).
8. **Stage events**: `starting` → `character_profiles` → `llm_calling` → ... → `saving` → `done` (`route.ts:185-308`).
9. **Build prompt**: `buildUserMessage(inp, references)` + `buildSystemPrompt()` (`route.ts:199-200`). Lihat §7.
10. **Heartbeat**: setInterval 2s, kirim `elapsedSec` (`route.ts:213-220`).
11. **LLM call**: `generatePromptPackage({provider, system, messages, onStreamChunk})` (`route.ts:222-229`). Stream chunks relayed via `stream_chunk` event (`route.ts:226-228`).
12. **LLM internal** (`llm-client.ts:237-424`):
    a. Decrypt API key (`llm-client.ts:244-254`).
    b. Build endpoint `${baseUrl}/chat/completions` (`llm-client.ts:256-257`).
    c. Request body: `{model, messages, max_tokens:32768, temperature:0.7, stream:true}` (`llm-client.ts:267-274`).
    d. fetch + AbortSignal.timeout(600_000) = 600s (`llm-client.ts:284-289`).
    e. Stream parse SSE: kumpulkan `delta.content` (`llm-client.ts:303-330`); fallback non-stream (`llm-client.ts:331-345`).
    f. `extractJsonFromContent(content)` (`llm-client.ts:353`) — strip `<think>`, code blocks, pick largest JSON (`llm-client.ts:106-165`).
    g. `JSON.parse(jsonStr)` first try (`llm-client.ts:358-360`); jika gagal → `repairTruncatedJson` + retry parse (`llm-client.ts:364-375`).
    h. `PromptPackageSchema.parse(parsedJson)` — Zod validation (`llm-client.ts:379`). Categorize error jika gagal (`llm-client.ts:388-394`).
    i. Loop retry: `maxRetries` default 2 (`llm-client.ts:238,279`). Backoff 2s/4s (`llm-client.ts:408-412`).
    j. Final throw `[CATEGORY] message` (`llm-client.ts:422-423`).
13. **LLM failure path** (`route.ts:238-264`): mark project `failed`, create generation log status `fail`, send `error` event `PROVIDER_ERROR`.
14. **Re-validate** (`route.ts:268-298`): `PromptPackageSchema.parse(pkg)` lagi di route. Jika fail → same failure path, event `VALIDATION_ERROR`.
15. **DB persist** (`route.ts:310-493`): updateProjectResult(status='complete') → delete old (image_prompts, supporting_chars, scenes, characters) → bulkCreateCharacters → bulkCreateScenes → per-scene createSceneAudio + scene image_prompts → bulk master image_prompts → bulk supporting_characters. Each wrapped `safeDbOp` (continue on error, `route.ts:35-51`).
16. **Consistency check**: `checkConsistency(validated)` → warnings[] (`route.ts:496`).
17. **Generation log persist**: status `partial` (jika warnings) atau `success` (`route.ts:502-513`).
18. **Done event**: `{result, warnings, generationLogId}` (`route.ts:518`).
19. **Unhandled catch** (`route.ts:520-548`): mark failed, log, send `PROVIDER_ERROR`.
20. **Finally**: close stream (`route.ts:549-552`).

export function tryExtractJson(text: string): unknown | null {
  const trimmed = text.trim();
  try { return JSON.parse(trimmed); }
  catch {
    const m = trimmed.match(/\{[\s\S]*\}/);   // ← greedy, ambil { pertama sampai } terakhir

### 8.2 `src/lib/ai/llm-client.ts` — extraction & repair (yang aktif)

#### 8.2.1 `extractJsonFromContent(content)` (`llm-client.ts:106-165`)

1. Strip `ILD...LD` blocks greedy (`llm-client.ts:110`).
2. Fallback jika `ILD` tanpa closing: cari real JSON via `findLargestJsonStart` (`llm-client.ts:113-131, 170-188`).
3. Strip `antml` tags (`llm-client.ts:134-135`).
4. Kumpulkan kandidat: code blocks triple-backtick json (`llm-client.ts:142-145`) + raw JSON via `extractAllJsonObjects` (`llm-client.ts:148, 193-212`).
5. Sort by length descending, pick largest (`llm-client.ts:157`).
6. Log candidate count + sizes (`llm-client.ts:159-162`).

#### 8.2.2 `repairTruncatedJson(jsonStr)` (`llm-client.ts:50-100`)

Strategi repair:
- Hapus trailing `"key":` tanpa value (`llm-client.ts:54`).
- Hitung unescaped quote, tutup string jika unterminated (`llm-client.ts:57-68`).
- Hapus trailing comma (`llm-client.ts:71`).
- Stack `{`/`[` dan tutup yang belum match (`llm-client.ts:74-97`).

**KETERBATASAN repair** (relevan Bug B):
- TIDAK handle **escape sequence rusak** di tengah string (mis. `\"` yang terpotong menjadi `\"` lalu newline).
- TIDAK handle **newline literal** di dalam string value (JSON standar melarang newline mentah dalam string; LLM kadang keluarkan newline mentah → parse fail).
- TIDAK handle **control char** (tab, `\t` mentah) di dalam string.
- TIDAK handle **trailing data setelah JSON valid** (jika LLM kirim `{...} teks tambahan`, parse fail meski JSON valid).
- TIDAK handle **duplicate key** (JSON.parse terima duplikat, ambil terakhir — tapi jika value type beda bisa lolos).
- Hanya close bracket/brace, tidak reconstruct value yang corrupt.

#### 8.2.3 Retry loop (`llm-client.ts:279-414`)

- `maxRetries` default **2** (`llm-client.ts:238`).
- Loop `attempt = 1..maxRetries` (`llm-client.ts:279`).
- Setiap attempt: fetch → stream → extract → parse → validate (`llm-client.ts:283-395`).
- Error categorization (`llm-client.ts:18-44`):
  - `TIMEOUT`: AbortError / "timeout" (`llm-client.ts:22-24`).
  - `NETWORK`: ECONNREFUSED/ENOTFOUND/fetch failed (`llm-client.ts:25-27`).
  - `VALIDATION`: ZodError / `.issues` (`llm-client.ts:28-36`).
  - `HTTP`: msg startsWith "Provider HTTP" (`llm-client.ts:37-39`).
  - `JSON_PARSE`: msg includes "JSON" or "parse" (`llm-client.ts:40-42`).
  - `UNKNOWN`: fallback (`llm-client.ts:43`).
- Backoff: `min(2000 * 2^(attempt-1), 8000)` = 2s, 4s (`llm-client.ts:409`).
- **PENTING**: retry TIDAK mengubah prompt/request. Body sama persis dikirim ulang (`requestJson` di-build sekali di `llm-client.ts:274`, fetch pakai `body: requestJson` `llm-client.ts:287`). LLM probabilistik (temp 0.7) → mungkin hasil beda, tapi jika prompt ambigu deterministik (seperti Bug A sfx_list), retry kemungkinan besar gagal sama persis.
- Final throw: `[CATEGORY] message` (`llm-client.ts:422-423`).

### 8.3 Validation trigger sequence

1. `llm-client.ts:379` `PromptPackageSchema.parse(parsedJson)` → throw ZodError jika mismatch (Bug A).
2. `route.ts:270` `PromptPackageSchema.parse(pkg)` (redundant — pkg sudah validated type, tapi parse ulang untuk safety).
3. Fail → `categorizeError` → category `VALIDATION` (`llm-client.ts:28-36`).

### 8.4 JSON parse failure sequence (Bug B)

1. `llm-client.ts:358` `JSON.parse(jsonStr)` throw SyntaxError.
2. `llm-client.ts:364-367` `repairTruncatedJson(jsonStr)` + `JSON.parse(repaired)`.
3. Jika repair gagal → throw `Response bukan JSON valid: <repairErr>` (`llm-client.ts:373`).
4. `categorizeError` lihat msg includes "JSON" → category `JSON_PARSE` (`llm-client.ts:40-42`).

## 11. Known Bug Analysis (root cause grounded in code)

### Bug A — VALIDATION: `sfx_list` Expected string, received array

**Gejala (log user)**: `scenes.N.audio_specs.2.sfx_list: Expected string, received array` di 3 scene. 2/2 attempt gagal → generate fail.

**Root cause (cited)**:

1. **Schema Zod mendefinisikan `sfx_list: z.string().nullable().optional()`** di `SceneAudioSpecSchema` (`src/lib/validation/schemas.ts:52`). Skema ini dipakai `SceneSchema.audio_specs` (`schemas.ts:63`). Validasi `PromptPackageSchema.parse` (`llm-client.ts:379`) reject array.

2. **Prompt ambigu tentang `sfx_list`** (`src/lib/ai/prompt-builder.ts:152`):
   ```
   AUDIO_SPECS: audio_type (background_music|sfx|ambient|music_cue|transition_audio), ... Untuk sfx: sfx_list.
   ```
   Tidak menyebut tipe data. Tidak menyebut "comma-separated string". Tidak beri contoh.

3. **Contoh JSON TIDAK pernah mendemokan `sfx_list`** (`prompt-builder.ts:75-97`). Hanya 2 audio_spec: `ambient` (tanpa sfx_list) dan `background_music` (tanpa sfx_list). LLM tidak punya demonstrasi format `sfx_list` sebagai string.

4. **Nama field `sfx_list`** secara linguistik menyiratkan "list" = array. LLM (MiniMax-M3) natural output: `["footstep","door creak","wind"]` alih-alih `"footstep, door creak, wind"`.

5. **DB column `scene_audio.sfxList: text`** (`schema.ts:193`) konsisten dengan schema Zod string. Jadi schema BENAR dari sisi storage, prompt yang salah dari sisi instruksi.

**Fix rekomendasi (untuk dokumen turunan, bukan implementasi di sini)**:
- Opsi 1 (prompt fix): ubah instruksi jadi eksplisit "sfx_list: string comma-separated, contoh 'footstep,door creak,wind'" + tambah audio_spec sfx di contoh JSON.
- Opsi 2 (schema fix): ubah `sfx_list: z.union([z.string(), z.array(z.string())]).nullable().optional()` lalu normalize di route sebelum DB insert (sudah ada pola ini untuk `color_palette` di `schemas.ts:29`).
- Opsi 3 (pre-validation transform): coerce array → string di route.ts sebelum `PromptPackageSchema.parse`.

**Rekomendasi RAG**: Opsi 2 paling robust (longgar di LLM, strict di DB). Tapi perlu normalizer di `route.ts:380-399` (audio save block) untuk `Array.isArray(audio.sfx_list) ? audio.sfx_list.join(', ') : audio.sfx_list`.

### Bug B — JSON_PARSE: malformed JSON at position 14719 line 246 col 100; auto-repair failed

**Gejala (log user)**: JSON malformed di retry, `repairTruncatedJson` gagal, 2/2 attempt fail.

**Root cause hypothesis (cited)**:

1. **Output panjang + `max_tokens:32768`** (`llm-client.ts:270`). Untuk tutorial (8-15 scene × 8-layer image prompts × 2 target), JSON mudah > 14719 char. MiniMax-M3 cenderung truncate atau salah escape di output panjang.

2. **`repairTruncatedJson` (`llm-client.ts:50-100`) keterbatasan** (lihat §8.2.2):
   - Tidak handle newline literal mentah di dalam string value (JSON standar larang). LLM sering keluarkan `\n` mentah di prompt_text panjang.
   - Tidak handle escape sequence rusak (`\"` terpotong).
   - Tidak handle control char (tab mentah).
   - Tidak handle trailing data setelah JSON valid.
   - Hanya tutup bracket, tidak reconstruct value corrupt.

3. **Retry TIDAK mengubah request** (`llm-client.ts:274,287`): `requestJson` di-build sekali, fetch pakai `body: requestJson` sama. Hanya `temperature:0.7` (`llm-client.ts:271`) yang kasih variabilitas. Jika bug deterministik (mis. LLM selalu salah escape karakter tertentu di posisi tertentu), retry gagal sama.

4. **`extractJsonFromContent` pick largest** (`llm-client.ts:157`) — jika ada JSON valid kecil + JSON rusak besar, pick yang rusak. Repair coba fix yang rusak, gagal.

5. **Position 14719 line 246 col 100** mengindikasikan: JSON panjang (~15KB+), error di tengah-bawah. Konsisten dengan scene ke-3+ audio_spec ke-3 (audio_specs.2 = index 2 = spec ke-3). Di retry, LLM output berbeda (temp 0.7) TAPI mungkin lebih panjang/truncate → JSON parse fail → repair fail → category JSON_PARSE → 2/2 fail. Artinya Bug A (prompt ambigu) memicu Bug B (output tidak stabil saat LLM dipaksa regenerate).

**Fix rekomendasi Bug B**:
- Tambah instruction eksplisit "Jangan gunakan newline mentah dalam string value, gunakan \\n escape" di prompt.
- Tingkatkan `repairTruncatedJson`: handle newline mentah (`replace(/\n/g, '\\n')` di dalam string terdeteksi), handle control char.
- Tambah pre-parse sanitizer: strip BOM, normalize line ending, escape control char sebelum `JSON.parse`.
- Tambah attempt dengan `stream:false` + `max_tokens:65536` jika attempt 1 stream truncate.
- Pertimbangkan provider/model lain yang lebih stabil di output panjang (ASUMSI: MiniMax-M3 via tokenrouter kurang stabil).
